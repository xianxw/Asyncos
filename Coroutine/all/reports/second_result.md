## 二、 性能对比深度分析

通过对上述基准测试数据的观察，可以清晰地印证操作系统理论中关于并发模型的核心差异。针对**网络爬虫这种典型的 I/O 密集型（I/O-bound）任务**，得出以下分析结论：

### 1. 吞吐率与总耗时：协程的微弱优势与多进程的劣势

* **数据现象：** 协程模型耗时最短（1.56 秒），吞吐率最高（21.21 req/s）；多线程紧随其后（1.62 秒）；而多进程表现最差，耗时增加了近 45%（2.26 秒）。
* **原理解释：** 在网络请求等待响应的过程中，CPU 大部分时间处于空闲状态。协程基于底层事件驱动（如 epoll 等 I/O 多路复用机制），在用户态进行轻量级的上下文切换，避免了系统调用开销，因此效率最高。在仅有 33 个并发量的情况下，OS 线程的上下文切换开销仍在可接受范围内，因此多线程表现与协程极为接近。而多进程的上下文切换涉及陷入内核态及页表刷新，开销过大，导致其在 I/O 任务中吞吐率显著下降。

### 2. 内存开销：多进程的性能陷阱与运行时的基础损耗

* **数据现象：** 多进程模型的内存飙升到了夸张的 673.47 MB，而多线程（38.48 MB）和协程（46.96 MB）均保持在极低的水平。值得注意的是，本次测试中多线程的内存占用甚至略低于协程。
* **原理解释：** 操作系统为每一个进程分配了独立的内存空间、堆栈、文件描述符等资源副本，33 个独立进程意味着基础资源被硬生生复制了 33 份，导致内存呈线性爆炸。相反，多线程和协程共享同一个进程的内存空间，因此占用极小。协程的内存略高于纯多线程，通常是因为引入 Rust 异步运行时（如 Tokio Reactor/Executor 框架及其任务队列）带来的固定基础内存开销，在并发基数较小（仅 33 个）时，这部分固定开销显得相对明显。

### 3. 延迟分布规律 (P50 与 P95)

* **数据现象：** 协程模型在 P50 (534.71 ms) 和 P95 (970.91 ms) 延迟上均表现最优。多线程模型的 P95 尾部延迟最严重 (1079.60 ms)，甚至劣于多进程。
* **原理解释：**
    * **协程的全局调度优势：** 协程调度由用户态的异步运行时统一调度，非阻塞机制使得所有准备好的网络响应能按最优顺序被处理，避免了无效挂起，因此拥有最稳定的延迟分布。
    * **多线程的抢占式惩罚：** 操作系统对线程进行抢占式调度。当几十个线程同时争抢 CPU 处理返回的网络数据包时，极易发生上下文颠簸，部分不幸的线程可能会被强制挂起较长时间，导致其对应的请求出现了长尾延迟（高达 1079.60 ms）。

### 4. 实验结论与工程建议

对于像网页抓取这样典型的 I/O 密集型任务，**异步协程（如使用 Rust 的 Tokio 生态）是综合表现最优的工程实践方案**。它在保证最高吞吐量和最低延迟的同时，内存占用也控制在极佳水平。多线程模型在并发量较低时（如本例的 33 并发）性能表现不俗，且代码心智负担较低，但在面临更高并发需求时，线程切换将逐渐成为瓶颈。多进程模式在此类场景下属于典型的反模式（Anti-pattern），白白浪费了大量内存资源，它应当只留给 CPU 密集型的重度计算任务使用。
